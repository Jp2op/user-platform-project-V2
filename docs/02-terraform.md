# Terraform

## Overview

All AWS infrastructure is defined in Terraform and runs exclusively from the GitHub Actions pipeline. The only exception is the one-time bootstrap which must be run locally.

```
terraform/
├── bootstrap/           ← Run once locally, creates state infra
│   ├── main.tf          ← S3 bucket, DynamoDB, OIDC provider, pipeline role
│   ├── variables.tf
│   ├── outputs.tf
│   └── terraform.tfvars
│
├── modules/             ← Reusable, environment-agnostic modules
│   ├── kms/             ← 4 KMS keys (EBS, RDS, Secrets, S3)
│   ├── vpc/             ← VPC, subnets, NAT, VPC endpoints
│   ├── eks/             ← Cluster, node group, addons, OIDC, EBS CSI IRSA
│   ├── rds/             ← RDS MySQL, Secrets Manager entries
│   ├── iam/             ← IRSA roles (ESO, ALB, Loki, GitHub deploy)
│   ├── s3/              ← Loki log storage bucket
│   ├── waf/             ← WAF v2 rules and logging
│   └── dns/             ← ACM certificate, Route53 records
│
├── main.tf              ← Root module — wires all modules together
├── variables.tf         ← All input variables with defaults
├── outputs.tf           ← Values needed for K8s layer and GitHub secrets
├── terraform.tfvars     ← Your specific values
└── backend.tf           ← S3 remote state (auto-generated by finish.ps1)
```

## Bootstrap

Bootstrap creates the foundation that every other Terraform run depends on. It is the **only time you run Terraform locally**.

### What Bootstrap Creates

| Resource | Purpose |
|----------|---------|
| S3 bucket | Stores Terraform state file remotely |
| DynamoDB table | Prevents concurrent terraform applies (state locking) |
| KMS key | Encrypts the state bucket and lock table |
| GitHub OIDC provider | Registers GitHub Actions as a trusted identity in AWS |
| Terraform Pipeline Role | IAM role that GitHub Actions assumes via OIDC |
| Developer Readonly Role | Lets developers inspect resources (no modify/delete) |

### Bootstrap Steps

```bash
# Prerequisites: AWS CLI configured, Terraform >= 1.7, Git

# 1. Fill in your values
cd terraform/bootstrap
# Edit terraform.tfvars:
#   github_org  = "YourGitHubUsername"
#   github_repo = "your-repo-name"

# 2. Run bootstrap
terraform init
terraform plan    # verify 14 resources, no errors
terraform apply   # type "yes"

# 3. Auto-generate backend.tf
.\finish.ps1      # Windows
# bash finish.sh  # Linux/Mac

# 4. Fill github_oidc_provider_arn in terraform/terraform.tfvars
# Use the value printed by finish.ps1

# 5. Add GitHub secrets
# repo → Settings → Secrets and variables → Actions → New secret
#   TF_PIPELINE_ROLE_ARN = <from finish.ps1 output>
#   AWS_REGION           = ap-south-1

# 6. Create GitHub environment
# repo → Settings → Environments → New → "production"
# Set deployment branch to "main" only

# 7. Commit and push — pipeline takes over
cd ../..
git add terraform/backend.tf terraform/terraform.tfvars
git commit -m "chore: configure terraform backend"
git push
```

### Why Pipeline-Only Execution

The state S3 bucket has a policy that denies writes from any principal except the pipeline role:

```
Developer runs terraform apply locally:
  → terraform plan succeeds (read-only)
  → terraform apply fails at state write (AccessDenied)

Pipeline runs terraform apply:
  → assumes TerraformPipelineRole via OIDC
  → terraform apply succeeds
```

This is enforced at the AWS level, not by convention. Even if someone has full AWS CLI access, they cannot modify infrastructure outside the pipeline.

## Module Architecture

### Dependency Order

The root `main.tf` calls modules in this order. Each module receives outputs from previous modules as inputs:

```
kms
  │ outputs: ebs_key_arn, rds_key_arn, secrets_key_arn, s3_key_arn
  ▼
vpc
  │ outputs: vpc_id, subnet_ids (public, private, isolated)
  ▼
eks
  │ outputs: oidc_provider_arn, cluster_oidc_issuer_url, node_sg_id
  │ creates internally: EBS CSI IRSA role (uses its own OIDC provider)
  ▼
s3                              rds
  │ outputs: loki_bucket_arn      │ outputs: qa_secret_arn, prod_secret_arn
  │                               │
  └───────────┬───────────────────┘
              ▼
            iam (depends_on: module.eks)
              │ outputs: role ARNs for ESO, ALB, Loki, GitHub deploy
              ▼
            waf (standalone)
              ▼
            dns (conditional — Route53 records only when alb_dns_name is set)
```

### Why EBS CSI IRSA Lives in the EKS Module

The EBS CSI addon needs an IRSA role. That role needs the EKS OIDC provider ARN. If we put the role in the IAM module:

```
eks needs ebs_csi_role_arn FROM iam
iam needs oidc_provider_arn FROM eks
= circular dependency — Terraform rejects this
```

By creating the EBS CSI IRSA role inside the EKS module, it uses the OIDC provider directly (same module, no cross-reference). All other IRSA roles stay in the IAM module because they're application-level concerns that don't cause cycles.

### Module Details

**kms** — Creates 4 separate KMS keys with dedicated key policies. Separate keys mean compromising one doesn't expose all encrypted data. Each key only grants access to its respective AWS service.

**vpc** — VPC with 3 subnet tiers (public, private, isolated) across 2 AZs. Single NAT Gateway for cost savings. VPC endpoints for S3 (gateway, free), ECR API, ECR DKR, Secrets Manager, and STS (interface, ~$7/month each). Interface endpoints keep AWS API traffic off the public internet and save NAT data transfer costs.

**eks** — EKS cluster with managed node group (ON_DEMAND, t3.medium). OIDC provider for IRSA. Managed addons: VPC CNI, CoreDNS, kube-proxy, EBS CSI driver, Pod Identity agent. EBS CSI IRSA role created internally to avoid circular dependency. Authentication mode is API_AND_CONFIG_MAP.

**rds** — Single RDS MySQL 8.0 instance (db.t3.micro) in isolated subnets. Two databases inside (qa_db, prod_db). Credentials auto-generated by Terraform, stored in Secrets Manager (3 secrets: qa, prod, root). Security group allows only EKS node SG on port 3306. Encrypted with dedicated KMS key. Automated backups with 7-day retention.

**iam** — All IRSA roles: ESO QA (reads only QA secret), ESO PROD (reads only PROD secret), ALB Controller (manages load balancers), Loki (writes to S3 bucket). GitHub Actions deploy roles: QA (branch-scoped to qa) and PROD (branch-scoped to main). EKS access entries grant kubectl access scoped to specific namespaces.

**s3** — Single bucket for Loki log storage. Versioned, KMS-encrypted, all public access blocked. Lifecycle rule deletes logs after 30 days.

**waf** — WAF v2 WebACL with 5 rule groups: IP Reputation List, Common Rule Set (OWASP Top 10), Known Bad Inputs, SQL Injection Rules, Rate Limiting (2000 req/5min per IP). Logs to CloudWatch for debugging blocked requests.

**dns** — ACM wildcard certificate (DNS-validated, auto-renewed). Route53 A records for app, www, and grafana subdomains. Records are conditional — only created when `alb_dns_name` is provided (after the K8s layer creates the ALB).

## Pipelines

### terraform.yaml — Plan & Apply

```
Trigger: push to main touching terraform/** or the workflow file

Job 1: Terraform Plan
  ├── Checkout
  ├── Assume AWS role via OIDC
  ├── terraform fmt -check
  ├── terraform init
  ├── terraform validate
  ├── terraform plan
  └── Post plan diff as PR comment (on PRs)
         │
         │ must succeed
         ▼
Job 2: Terraform Apply
  ├── Requires: "production" environment approval
  ├── Checkout
  ├── Assume AWS role via OIDC
  ├── terraform init
  ├── terraform apply -auto-approve
  └── Print outputs
```

### terraform-destroy.yaml — Destroy Infrastructure

```
Trigger: manual only (workflow_dispatch)
Confirmation: must type "DESTROY" exactly

Job 1: Validate confirmation

Job 2: Destroy
  ├── Scale EKS nodes to 0 (speeds up node group deletion)
  ├── Empty S3 buckets (handles versioned objects + delete markers)
  ├── Clean orphan VPC resources (subnets/SGs from failed partial applies)
  └── terraform destroy -auto-approve
```

The destroy pipeline handles three common failure scenarios:
- **EKS node group timeout** — scales to 0 first so instances terminate before the group is deleted
- **S3 BucketNotEmpty** — empties all versioned objects before Terraform tries to delete the bucket
- **VPC DependencyViolation** — finds and deletes orphan subnets, security groups, and VPC endpoints that aren't tracked in Terraform state (typically from cancelled partial applies)

## Customisation

To reuse this for a different project, change only `terraform/terraform.tfvars`:

```hcl
project_name = "myapp"
cluster_name = "myapp-cluster"
domain_name  = "myapp.com"
github_org   = "myusername"
github_repo  = "my-repo"
```

To add a third AZ (more resilient, no code changes):

```hcl
public_subnet_cidrs   = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
private_subnet_cidrs  = ["10.0.10.0/24", "10.0.11.0/24", "10.0.12.0/24"]
isolated_subnet_cidrs = ["10.0.20.0/24", "10.0.21.0/24", "10.0.22.0/24"]
```

To switch node capacity:

```hcl
node_capacity_type = "SPOT"       # ~70% cheaper, can be slow
node_capacity_type = "ON_DEMAND"  # instant, full price
```

## Outputs Reference

After `terraform apply` completes:

| Output | Where To Use It |
|--------|----------------|
| `kubeconfig_command` | Run locally to connect kubectl |
| `certificate_arn` | ALB ingress annotation for HTTPS |
| `waf_acl_arn` | ALB ingress annotation for WAF |
| `github_qa_deploy_role_arn` | GitHub secret: `AWS_ROLE_TO_ASSUME` |
| `github_prod_deploy_role_arn` | GitHub secret: `AWS_ROLE_TO_ASSUME_PROD` |
| `alb_controller_role_arn` | K8s bootstrap: ALB controller service account |
| `eso_qa_role_arn` | K8s bootstrap: ESO QA service account |
| `eso_prod_role_arn` | K8s bootstrap: ESO PROD service account |
| `loki_role_arn` | K8s bootstrap: Loki service account |
| `rds_endpoint` | Already in Secrets Manager, app reads via ESO |
